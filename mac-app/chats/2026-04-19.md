## First-launch onboarding + UserMode (supply vs full-experience)

**Branch:** `taylorhou/onboarding-supply-mode`

### Goal
"Super simple for anyone to download, install, onboard, and supply, with the option to use as well but not chat." Install pipeline was already production-grade (Developer-ID signed + notarized release via GitHub Actions, `curl | sh` one-liner). The gap was the first-launch experience: users landed directly in ChatView with no welcome, no "pick a supply model," and no way to hide the chat UI when they only wanted to earn.

### What shipped

**New `UserMode` enum** (`Sources/AppCore/AppState.swift`):
```swift
public enum UserMode: String, CaseIterable, Hashable, Sendable, Codable {
    case supplyOnly    // earn only, no chat UI
    case supplyAndAPI  // supply + local OpenAI API, no chat UI
    case full          // supply + API + chat
}
```
All three modes supply compute — the only axis is whether the chat surface is shown. `supplyAndAPI` is for devs using their Mac as a backend endpoint without needing the chat window.

**`AppState` additions:**
- `userMode: UserMode` — persisted to `teale.userMode` UserDefault
- `onboardingCompleted: Bool` — persisted to `teale.onboardingCompleted`, defaults to `true` for existing installs (gated on presence of `teale.stable_node_id` so returning users never see the welcome sheet)
- `currentView` default is now computed: `.chat` when chatEnabled, `.dashboard` otherwise

**New `WelcomeView`** (`Sources/InferencePoolApp/Views/WelcomeView.swift`):
- 3 mode cards with icons, subtitles, and a "Recommended" chip
- Recommendation keyed on RAM: ≥64 GB → full, ≥24 GB → supply+API, else supply-only
- Continue button is disabled until a card is picked; card selection is idempotent
- Sheet is `.interactiveDismissDisabled()` so users can't escape without choosing
- On confirm: writes `userMode`, sets `contributeCompute = true`, sets `currentView`, and flips `onboardingCompleted`

**Sidebar gating** (`InferencePoolApp.swift`):
- Chat entry in `MainSidebar` only rendered when `userMode.chatEnabled`
- `detailContent` switch falls back to `DashboardView()` if `.chat` is selected but chat is disabled (belt-and-suspenders for deep links or saved state)
- WelcomeView presented as `.sheet(isPresented:)` over `MainWindowView`, bound to `!onboardingCompleted`

**Settings affordance** (`Sources/InferencePoolApp/Views/SettingsView.swift`):
- Added an "Experience" Picker below the existing "Contribute compute" toggle so users can switch modes post-onboarding
- Per-mode blurb explains what each mode does

### Network-hardening diagnostic (done, no change shipped)
Before starting the UX work, checked `gateway.teale.com/v1/models`: only 1 model visible (`meta-llama/llama-3.1-8b-instruct`) despite 8 devices connected. Root cause per `gateway_devices_eligible` metric: the other 3 catalog models (`hermes-3-llama-3.1-8b`, `deepseek/deepseek-v3.2`, `qwen/qwen3.6-35b-a3b`) each have only 1 loaded node, under the small-floor of 2. A fourth entry (`C:/Teale/models/Qwen3-4B-Q4_K_M.gguf`) is a stale pre-openrouterId build advertising a raw Windows filepath — will rot off when that node upgrades. Fix path: grow supply organically via the onboarding ship; do NOT lower the floor (OR wants demonstrated reliability, and showing a model with only one supplier would be a liability).

## Follow-up: auto-download starter model on welcome confirm

**Branch:** `taylorhou/onboarding-model-autodownload`

PR #12 closed the "pick a mode" gap but left a second cliff: a supply-only user with no model loaded is supplying nothing. This follow-up wires the welcome's Continue button to kick off a background download of the top catalog model that fits the user's RAM. `AppState.downloadModel` auto-loads on completion when `engineStatus == .idle` (already there), so by the time the dashboard is visible the model is downloading with progress; when it finishes, the engine loads it and supply begins.

### Guard: only for brand-new installs
The download only triggers when all three of `downloadedModelIDs`, `activeDownloads`, and `scannedGGUFModels` are empty — any signal that the user already has a model means we skip. This keeps the auto-download from touching returning users who came through the settings picker to re-run onboarding, or users who sideloaded a GGUF before clicking Continue.

### Starter model choice
`modelManager.catalog.topModels(for: hardware, limit: 1).first` — `llama-3.1-8b-instruct-4bit` has `popularityRank: 1` in ModelCatalog.swift, so on any Mac with ≥8 GB RAM (which is every supported Mac) this resolves to Llama 3.1 8B Q4. Its ~4.5 GB download also happens to be the most-in-demand model in the fleet, which is what we want: a one-click install that grows the OpenRouter-facing supply pool.

### UI note on the welcome
Added a caption line above the existing "You can change this any time in Settings" hint: "We'll start downloading \(name) (\(X) GB) in the background." The sentence only shows when the download is going to fire — if the user already has a model it's hidden.

## Follow-up: gate the starter download on an explicit checkbox

**Branch:** `taylorhou/welcome-download-opt-in`

User feedback on PR #17: auto-downloading 4.5 GB on Continue is not something we should do without explicit permission. Converted the caption-line-that-foreshadows-an-implicit-download into a **Toggle with `.checkbox` style**, defaulted OFF, above the Continue button.

Copy on the checkbox: "Download \(name) now (\(X) GB)" with a secondary caption "Needed to supply compute. You can skip and pick a model later in Models." That keeps the door open to one-click supply for users who *want* the model now, while making the permission explicit. The checkbox only renders when the auto-download would be applicable (none of downloadedModelIDs, activeDownloads, scannedGGUFModels is populated) — returning or sideload users don't see it at all.
