# 2026-04-22 — Restore Google sign-in for 200-machine partner

## What happened

First partner company (200 machines, multi-platform) primarily authenticates via Google
Workspace OAuth. Prior test failed with a specific symptom: Google auth page loaded but
the email input was unusable — pointing at a Google Cloud Console / Supabase dashboard
misconfiguration, not the Swift code.

Separately, the Google button had been dropped from `LoginView` during the April 18 flatten
(commit `f96ddf4`), so even a correctly configured backend had no UI entry point on mac-app.

## Change

`mac-app/Sources/AuthKit/Views/LoginView.swift` — restored the Google OAuth button above
the existing GitHub button. Wired to the already-working `handleOAuth(provider: .google)`
path. No `AuthManager` change needed — `signInWithOAuth(provider:)` at `AuthManager.swift:168`
already passes `provider` through to the Supabase SDK.

No `options` / `hd` param: user confirmed any Google account (personal + Workspace) should be
allowed, so no hosted-domain restriction.

## Out of scope

- Windows, Linux, Android: no Supabase client in repo today. Flagged as separate workstream
  so the Windows workspace picking this up knows what Mac has and what's missing elsewhere.
- Apple Sign In: still off (DUNS reason from commit `3bf8aaa`).

## Dashboard setup (external, not code)

User completed these on their side:

- Google Cloud Console → OAuth consent screen: **External** + published (not Internal).
- Google Cloud Console → Credentials → OAuth 2.0 Client (Web application) with authorized
  redirect URI `https://<project_ref>.supabase.co/auth/v1/callback`.
- Supabase dashboard → Auth → Providers → Google: enabled with matching client ID/secret.
- Supabase dashboard → Auth → URL Configuration: `teale://auth/callback` added to the
  redirect allow-list.

## Build

`swift build` — 104.16s, clean (only pre-existing `stub` warning in `InferencePoolApp.swift`).
