# 2026-04-15: Darkbloom Competitive Analysis

## Context

Darkbloom.dev launched — a competing decentralized inference network on Apple Silicon built by Eigen Labs (the EigenLayer team). They published a paper: [Private Decentralized Inference on Consumer Hardware](https://github.com/Layr-Labs/d-inference/blob/master/papers/dginf-private-inference.pdf).

Their key innovation: Mac owners providing inference **provably cannot see user prompts or responses** via "Software Access Path Elimination."

## Darkbloom Overview

- ~14K lines of code (Rust, Go, Swift, TypeScript, Python)
- OpenAI-compatible API, pay-per-token, 0% platform fee
- Central coordinator running in Intel TDX hardware TEE
- Provider agent is a hardened Rust binary with embedded MLX inference via PyO3
- **Proprietary license** ("Eigen Labs, Inc. All rights reserved")

## How Their Private Inference Works

They eliminate every software path through which a Mac owner could observe inference data:

1. **In-process inference**: MLX engine runs directly inside a single hardened Rust process via PyO3 FFI. No subprocess, no localhost HTTP server, no IPC.
2. **PT_DENY_ATTACH**: Kernel permanently denies all ptrace requests (lldb, dtrace, Instruments) — even from root.
3. **Hardened Runtime**: Binary signed without `get-task-allow` entitlement. Kernel denies `task_for_pid()` and `mach_vm_read()` from any external process.
4. **SIP Immutability (Theorem 1)**: Formally proven that SIP, once verified at process startup, cannot be disabled during process lifetime — because disabling requires a reboot, which kills the process.
5. **Hypervisor memory isolation**: `Hypervisor.framework` Stage 2 page tables make inference memory invisible to RDMA/DMA. 0% performance overhead.
6. **Memory sanitization**: Prompt/response buffers zeroed with `write_volatile` after each request.

Residual attack: physically desoldering LPDDR5x memory from the SoC (destructive). Same threat model as Apple Private Cloud Compute.

## Their 4-Layer Attestation

| Layer | Mechanism | What It Proves |
|-------|-----------|---------------|
| 1 | Secure Enclave P-256 ECDSA | Hardware-bound identity, replay prevention |
| 2 | MDM SecurityInfo | Independent SIP/SecureBoot verification from OS (not provider software) |
| 3 | Apple MDA (ACME device-attest-01) | Apple-signed cert chain proving genuine hardware, serial, UDID |
| 4 | Challenge-response (every 5 min) | Liveness, security posture hasn't degraded |

## Teale vs Darkbloom Comparison

| | Teale | Darkbloom |
|---|---|---|
| **Can Mac owner see prompts?** | Yes — node decrypts plaintext for inference | No — provably blocked via access path elimination |
| **Architecture** | Fully P2P, no central coordinator | Hub-and-spoke, central coordinator in Intel TDX TEE |
| **License** | AGPL-3.0 (fully open source) | Proprietary ("All rights reserved") |
| **Enrollment friction** | Zero — just run it | MDM profile installation required |
| **Provider identity** | Ed25519 (software) | P-256 in Secure Enclave (hardware-bound) |
| **Hardware attestation** | None | Apple MDA certificate chain |
| **Encryption** | Optional Noise protocol (transit only) | E2E NaCl Box + process memory isolation |
| **Network topology** | 4 tiers (local/LAN/PTN/WWTN) | Single global tier with trust levels |
| **Consumer interface** | Native iPhone app | OpenAI-compatible HTTPS API |
| **Platform fee** | 1.8% | 0% |
| **Single point of failure** | None | Coordinator |

## Where Darkbloom is Stronger

- **Private inference from untrusted providers** — rigorous formal proof, complete attack surface enumeration
- **Apple-level security model** — claims parity with Apple Private Cloud Compute
- **Hardware attestation** — Apple MDA certificates are unforgeable without compromising Apple itself
- **Negligible overhead** — 12ms per request security checks, 0% hypervisor overhead
- **Drop-in API** — OpenAI-compatible, lower adoption friction for developers

## Where Teale is Stronger

- **Fully open source (AGPL-3.0)** — Darkbloom is proprietary. Their paper describes standard macOS APIs (PT_DENY_ATTACH, Hardened Runtime, Hypervisor.framework, Secure Enclave CryptoKit, MDM SecurityInfo). Nothing is patented. The community can implement these techniques on Teale's open P2P architecture.
- **No central coordinator** — no single point of failure, no entity to subpoena or shut down, no metadata visibility
- **Zero enrollment friction** — no MDM profiles on personal machines
- **Private TealeNet (PTN)** — organizational subnets with certificate-based membership
- **Multi-tier network** — local/LAN/PTN/WWTN with WFQ prioritization
- **Native consumer UX** — iPhone app vs API-only

## Darkbloom's Weaknesses

1. **Proprietary license** — code is visible on GitHub but not forkable or modifiable
2. **Central coordinator dependency** — Intel TDX TEE still sees routing metadata; OHTTP anonymization listed as "future work"
3. **MDM enrollment friction** — many Mac owners will refuse MDM on personal devices
4. **Kernel zero-day assumption** — entire security model collapses on a macOS kernel vulnerability (they acknowledge this)
5. **Not audited** — repo states "under active development and has not been audited"
6. **Timing side channels** — token timing reveals prompt/response length; no mitigation implemented
7. **Python GIL** — PyO3 embedding serializes to one request at a time; Rust-native MLX FFI is prototype only

## Strategic Takeaway

The privacy gap is real but primarily matters for WWTN (routing to strangers). For local/LAN/PTN, users trust their own devices or known peers.

Teale's open-source advantage is decisive here: Darkbloom published a blueprint using standard, publicly documented macOS APIs. The community can implement equivalent privacy hardening on Teale's superior P2P architecture — without the centralized coordinator, MDM friction, or proprietary restrictions.

We build in the open. They don't.

# Session 2: Device-Model Profiles

## Context

Models loaded with default settings don't work well across devices. The existing RAM-based tuning in AppState.swift was just 2 thresholds (64GB, 128GB) controlling 4 parameters. Different (device, model) combinations need specific context window sizes, KV cache quantization, batch sizes, flash attention settings, etc.

## What Was Built

**Device-model profile system** — a registry that maps `(device class, model)` → optimal llama.cpp inference parameters.

### New Files
- `Sources/SharedTypes/DeviceModelProfile.swift` — `DeviceClass` enum (5 classes from 21+ chips), `InferenceProfile` struct (all tunable params), `DeviceModelProfile` struct, `ChipFamily.deviceClass` extension
- `Sources/HardwareProfile/DeviceModelProfileRegistry.swift` — ~25 curated profiles covering all device classes with RAM-conditional and model-specific overrides
- `Sources/HardwareProfile/DeviceModelProfileResolver.swift` — fallback chain resolver: user override → model ID → model family → device-class default → global default. Post-processes flash attention for M1/M2 (generation < 3).
- `Sources/HardwareProfile/UserProfileOverrides.swift` — loads/saves user overrides from `~/.teale/profiles.json`

### Modified Files
- `Sources/AppCore/AppState.swift` — replaced hardcoded RAM thresholds with `DeviceModelProfileResolver.resolve()` at init time; added model-specific profile application in `loadGGUFModel()` via `updateConfiguration()` before each model load

### Key Design Decisions
- **5 device classes** (ultraDesktop, maxDesktop, proLaptop, baseMac, mobile) instead of per-chip profiles — avoids combinatorial explosion
- **Layered merge**: profiles only specify fields they want to override; nil fields inherit from lower layers
- **User overrides first**: `~/.teale/profiles.json` takes priority over built-in profiles — device owners can always override recommendations
- **Demand-driven**: `ModelDemandTracker` tells you *what* to run; profiles tell you *how* to run it well. No changes needed to demand tracker.
- **Flash attention gate**: M1/M2 chips get `flashAttn: false` regardless of profile, since M3+ has better hardware support
