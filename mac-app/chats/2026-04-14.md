## Session 4: WAN Reliability Fixes — Drop Tailscale

### What was done

**Five WAN reliability fixes (PR #42):**

1. **WebSocket keepalive** — 25-second ping loop prevents NAT/firewall idle timeout from killing the relay WebSocket. Previously, the WebSocket would die after 30-60s of idle.

2. **Relay session suspension/recovery** — on WebSocket disconnect, relay sessions are preserved (not destroyed). After reconnect, sessions are re-opened on the new WebSocket. Previously, ALL relay peer connections were killed on any WebSocket hiccup, and both sides would race to reconnect.

3. **handleIncomingOffer relay fallback** — returns nil instead of throwing when direct connection fails for symmetric NAT. The initiator's relay fallback (relayOpen → relayReady) is handled by handleIncomingRelayOpen, so the responder just needs to not crash.

4. **NoiseSession lifetime** — extended from 3 minutes to 24 hours. The 3-minute expiry was a ticking time bomb for long-lived connections.

5. **Reconnect deduplication** — `reconnectingNodeIDs` set prevents multiple reconnect tasks for the same peer. Heartbeat failure tolerance increased from 3 to 12 during relay reconnection.

**Deployment & Testing:**
- Built and deployed latest Teale to both Mac Studio and Mac Mini
- Mac Studio: GGUF model auto-loaded, WAN enabled, relay connected
- Mac Mini: WAN connected to Mac Studio via relay (both symmetric NAT → relay-only)
- **End-to-end WAN inference confirmed:** Mac Mini localhost:11435 → WANProvider → relay → Mac Studio → llama-server → Qwen3-32B-Q8_0 → response back through relay
- Hermes config updated: `base_url: http://localhost:11435/v1` (routes through local Teale WAN, NOT direct Tailscale IP)

### Key Decisions
- **Relay-only for symmetric NAT is fine** — both machines have symmetric NAT so direct P2P is impossible. The relay adds ~50-100ms latency but is reliable with the keepalive fix.
- **Hermes now routes through local Teale** at `localhost:11435` instead of direct Tailscale IP `100.88.240.125:11435`. This means Tailscale is no longer needed for inference routing.

### PRs Merged
- PR #42: Fix WAN reliability (keepalive, relay session recovery, symmetric NAT)

## Session 5: TealeNet Architecture — PTN, WFQ, WWTN Auction

### What was done

**PR #44 — TealeNetKit: PTN Membership System**
- New TealeNetKit module: Ed25519 CA certificates for Private TealeNet membership
- PTNCertificateAuthority: generates CA keypair per PTN, signs membership certificates
- PTNCertificate: payload + Ed25519 signature, P2P verification against CA public key
- PTNInvite: base64url-encoded invite tokens with nonce and expiry
- PTNStore: file persistence under ~/Library/Application Support/Teale/ptn/
- PTNManager: create PTN, generate invites, handle join requests, verify certs
- NodeCapabilities gains ptnIDs field for WAN discovery broadcasting
- Settings UI: create PTN, join via invite code, manage memberships with role badges

**PR #45 — Electricity Cost Floor Pricing**
- HardwareCapability.estimatedInferenceWatts: power draw per chip (M1 20W → M4 Ultra 75W)
- InferencePricing.electricityFloorCost(): min charge from watts × time × $/kWh
- InferencePricing.effectiveCost(): max(token price, electricity floor) — providers never earn below electricity cost + 20% margin
- Settings UI: $/kWh input with currency picker, device power draw display

**PR #46 — PTN Relay Exchange, WFQ Scheduler, WWTN Auction**
- PTN invite exchange over WAN relay: join request → signed certificate response
- PTN-aware WAN routing: connectedPeerForInference uses ptnIDs array
- RequestScheduler: Weighted Fair Queuing with PTN (0.7) and WWTN (0.3) weights. When one queue empty, other gets 100% — zero idle capacity. Within WWTN: highest-bid-first.
- WWTNAuction: Akash-style reverse auction. Providers auto-bid their floor price. Requestors post max bid. Lowest bid wins. Floor derived from electricity cost.

### Architecture Decisions
- **4 inference tiers:** Local → LAN Cluster → PTN (Private TealeNet) → WWTN (World Wide TealeNet)
- **PTN membership:** Certificate-based, not request/approve. Invite = CA signs joiner's cert. Fully P2P verification.
- **Pricing:** PTN = fixed rate table. WWTN = reverse auction with electricity floor.
- **Prioritization:** WFQ, not capacity reservation. Zero idle RAM.
- **Electricity is the real marginal cost** — hardware depreciation is nominal. Floor price = watts × time × $/kWh.

### PRs Merged
- PR #44: TealeNetKit PTN membership
- PR #45: Electricity cost floor pricing
- PR #46: PTN relay exchange, WFQ scheduler, WWTN auction

- Ed25519 identity compatible with Swift's `WANNodeIdentity` format
- WebSocket relay client matching the Swift `RelayClient` protocol exactly
- ClusterMessage parsing: hello/helloAck, heartbeat/heartbeatAck, inference flow
- llama-server subprocess management with health checks

**Docker Distribution:**
- `Dockerfile` — NVIDIA CUDA runtime + llama-server + teale-node
- `Dockerfile.cpu` — CPU-only variant (Debian slim)
- `scripts/build-all.sh` — multi-platform build script
- `teale-node.example.toml` — sample configuration

**Regression Verification:**
- `swift build` passes (3.01s, no recompilation needed)
- All existing `HardwareCapability` callsites still work (optional params with defaults)
- No changes to `InferenceProvider` protocol
- No changes to relay server

### Key Decisions
- **Rust over Swift for cross-platform:** Swift on Linux is heavy, Windows immature, Android unsupported. Rust gives single static binary for every target.
- **HTTP subprocess approach over native C bindings:** Matches existing `LlamaCppProvider` pattern, proven reliable, avoids linking complexity
- **Plaintext relay first:** Swift side already has plaintext fallback in `RelayPeerConnection.swift`. Noise encryption is Phase 5.
- **Optional fields for backward compat:** Existing Apple nodes decode new fields as nil, no breakage

### Architecture
```
teale-node (Linux/Windows/Android) → WebSocket → Relay Server ← WebSocket ← Mac/iPhone Teale App
     ↓ HTTP (localhost)
 llama-server (GGUF model, CUDA/ROCm/Vulkan/CPU)
```

## Session 8: Fix Qwen3 Hallucination on Mac Studio (tailor512)

### What was done

**Root cause:** LlamaCppProvider forced `--rope-scaling yarn --yarn-orig-ctx 40960 --override-kv qwen3.context_length=int:131072` on every model when `contextSize > 40960`. On the 512GB Mac Studio (`contextSize = 131072`), this always fired. Qwen3-235B-A22B already has YaRN baked into its GGUF weights — double-applying it with a wrong original context length (40960 vs actual training context) corrupted positional embeddings, producing completely incoherent "drunk" output.

**Fixes:**
1. **Removed forced YaRN block** from `LlamaCppProvider.startServer()` — llama-server reads RoPE config from GGUF metadata automatically. No external rope scaling needed.
2. **Reduced default context sizes** — `128GB+ RAM: 131072 → 65536`, `<128GB: 65536 → 32768`. Stays within most models' native context; models with longer native context still work via GGUF metadata.

### Files Changed
- `Sources/LlamaCppKit/LlamaCppProvider.swift` — removed YaRN scaling block
- `Sources/AppCore/AppState.swift` — conservative context size defaults

### Key Decisions
- **Let llama-server handle RoPE automatically** — GGUF files contain the correct rope config for each model. External overrides are fragile and model-specific.
- **`--reasoning off` kept as-is** — not the cause of incoherence, just suppresses `<think>` tags.
