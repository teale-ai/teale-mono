# 2026-04-07 — WAN Same-Network Test Prep

## What was done

- Investigated the current WAN path before testing same-network peer connectivity outside the LAN cluster.
- Confirmed two functional gaps:
  - discovered WAN peers were never auto-connected
  - loaded model state was not propagated to WAN discovery/heartbeats after startup
- Patched the WAN client to:
  - auto-connect newly discovered WAN peers from relay discovery
  - update advertised WAN capabilities when a model is loaded or unloaded
  - include a preferred non-tunnel local IPv4 address in WAN offer/answer payloads
  - prefer that local IPv4 path when two peers appear to share the same public IP
- Explicitly excluded tunnel-style interfaces such as `utun` from the preferred local IP selection so Tailscale does not become the chosen direct path for same-LAN WAN tests.
- Synced the changes to `CDPO16`, rebuilt the app bundle there, and relaunched the updated app.

## Key decisions

- Kept the test on the WAN stack rather than Bonjour/LAN clustering by using relay-based discovery and direct QUIC connection setup.
- Added same-public-IP local-address preference because same-network WAN testing is otherwise unreliable when both peers try to connect via the router’s public/NAT-mapped address.
- Did not rely on Tailscale for this path; the goal was same-LAN direct connectivity initiated via WAN discovery/control plane.

## Verification

- `swift build` succeeded locally after the WAN changes.
- Remote rebuild on `CDPO16` succeeded with the updated WAN code.
- Updated remote app launched successfully and is running as `InferencePoolApp`.

# 2026-04-07 — Persistence Regression Fixes

## What was done

- Reviewed the `2026-04-06` commit history and traced the persistence regressions to the large UX/auth commit `11a05d1`.
- Fixed the Settings version display so it no longer uses a hardcoded string.
- Updated `bundle.sh` to stamp the built app bundle with a date-based version and build number:
  - `CFBundleShortVersionString = YYYY.MM.DD`
  - `CFBundleVersion = YYYYMMDDHHMM`
  - `TealeBuildDate = YYYY.MM.DD`
- Persisted `maxStorageGB` in `UserDefaults` and wired it to update `ModelCache` instead of keeping it as a settings-only temporary value.
- Replaced the launch-random auth device ID with a stable per-install ID stored in `UserDefaults`.
- Updated auth device registration to:
  - fall back to matching by device name + platform when the stored ID is missing or changed
  - update an existing row instead of inserting another one
  - deactivate extra duplicate rows found for the same logical device
  - deduplicate device rows client-side by preferring the most recently seen entry
- Synced these fixes to `CDPO16`, rebuilt the app, and relaunched it.

## Root causes

- The Settings version label was hardcoded to `0.1.0` in the app UI.
- The storage slider only changed local `@State`; it never persisted to disk and never updated `ModelCache`.
- Device registration used a fresh random UUID on each launch before WAN was enabled, so every relaunch looked like a new device.

## Verification

- `swift build` succeeded locally after the fixes.
- Local `Teale.app` bundle now reports:
  - `CFBundleShortVersionString = 2026.04.07`
  - `CFBundleVersion = 202604071510`
  - `TealeBuildDate = 2026.04.07`
- Remote `CDPO16` bundle now reports:
  - `CFBundleShortVersionString = 2026.04.07`
  - `CFBundleVersion = 202604071511`
  - `TealeBuildDate = 2026.04.07`
- Updated remote app launched successfully and is running as `InferencePoolApp`.

# 2026-04-07 — Remote Control API

## What was done

- Added a dedicated remote-control API surface to `LocalAPI` instead of overloading the inference endpoints.
- Added new endpoints:
  - `GET /v1/app`
  - `PATCH /v1/app/settings`
  - `POST /v1/app/models/download`
  - `POST /v1/app/models/load`
  - `POST /v1/app/models/unload`
- Added `RemoteControlBridge` in the app target so the HTTP server can safely manage `AppState` for:
  - model download/load/unload
  - WAN toggle
  - LAN cluster toggle
  - relay URL
  - cluster passcode
  - org capacity reservation
  - max model storage
- The remote model operations now propagate real errors instead of silently swallowing failures.
- Synced the new control surface to `CDPO16`, rebuilt the app, and relaunched it.

## Important limitation

- Remote control only works once the Teale local server is running on the target machine.
- This patch does not add an out-of-band daemon or launch agent, so you still need one live server process to accept the API calls.

## Verification

- `swift build` succeeded locally after adding the management endpoints.
- Remote rebuild on `CDPO16` succeeded with the new API code.
- Relaunched remote app is running as `InferencePoolApp`.
