# 2026-04-05 — Initial Build Session

## What was done

### Phase 1: Local Inference App (from scratch)
- Scaffolded entire Swift package with 7 modules from empty directory
- Built SharedTypes: InferenceProvider protocol, OpenAI-compatible API types, hardware types, model descriptors, engine status, device registration stubs
- Built HardwareProfile: chip/RAM/GPU detection via sysctl + IOKit, thermal/power/activity/network monitors
- Built MLXInference: MLXProvider actor wrapping mlx-swift-lm, custom HFDownloader + HFTokenizerLoader adapters (bridging swift-transformers to MLXLMCommon protocols)
- Built ModelManager: curated catalog of 10 models (1B-70B), ModelCache actor for ~/Library/Application Support, ModelManagerService facade
- Built InferenceEngine: AdaptiveThrottler (battery/thermal/activity-aware), InferenceEngineManager actor
- Built LocalAPI: Hummingbird HTTP server on localhost:11435 with /v1/chat/completions (streaming SSE + non-streaming) and /v1/models
- Built InferencePoolApp: MenuBarExtra SwiftUI app with dashboard, chat UI, model browser, settings
- In-memory conversation persistence (SwiftData requires Xcode SDK)

### Phase 2: LAN Cluster Mode
- Created ClusterKit module (13 files)
- Wire protocol: ClusterMessage Codable enum with hello, heartbeat, inference, model sharing message types
- Transport: NWProtocolFramer (4-byte length prefix + JSON), PeerConnection async wrapper, passcode security
- Discovery: BonjourService (NWListener + NWBrowser on _inferencepool._tcp), PeerResolver handshake
- Cluster core: ClusterManager orchestrator, PeerInfo, PeerHealthMonitor (5s/15s/30s), ClusterTopology sorted by capability
- Routing: RequestRouter picks best node, ClusterProvider implements InferenceProvider for cluster delegation
- Model sharing: ModelSharingService with 1MB chunked file transfer
- Refactored InferenceEngineManager to accept `any InferenceProvider` (no longer hardcodes MLXProvider)
- Added ClusterView, PeerCardView, cluster settings section, dashboard cluster summary

## Key decisions
- Used swift-transformers Hub module directly instead of swift-hf-api-mlx adapters (dependency conflict with branch-based mlx-swift-lm)
- In-memory persistence instead of SwiftData (SwiftData macros need Xcode SDK, not available with Command Line Tools)
- Hummingbird 2.x over Vapor (lighter weight) and raw SwiftNIO (too low-level)
- Custom NWConnection protocol over HTTP for inter-node communication (lower latency for heartbeats + streaming)
- MVP cluster uses full-model request routing, not pipeline parallelism (stretch goal)

## Dependency resolution issues encountered
- mlx-swift-examples doesn't export MLXLLM (moved to mlx-swift-lm repo)
- swift-hf-api-mlx / swift-tokenizers-mlx conflict with branch-based mlx-swift-lm dependency
- Solved by writing own Downloader + TokenizerLoader implementations using swift-transformers directly
- NWTXTRecord API differences from documentation (getEntry uses `for:` not `key:`)
- Nested struct memberwise inits are internal across modules — added explicit public inits

## Build status
- `swift build` succeeds (86MB arm64 binary)
- Tests require Xcode SDK (XCTest not available with Command Line Tools only)
- Full app experience (MenuBarExtra) requires opening Package.swift in Xcode

# 2026-04-05 — Session 2: Phases 3, 4, 5

## What was done

### Phase 3: WAN P2P Networking
- Created WANKit module (8 files) for internet-scale P2P networking
- WANNodeIdentity: Ed25519 keypair via CryptoKit, Keychain persistence, hex-encoded node IDs
- RelayClient: WebSocket-based signaling server client with exponential backoff reconnection
- STUNClient: RFC 5389 STUN binding requests over UDP, NAT type detection (full cone, restricted, symmetric)
- QUICTransport: Network.framework native QUIC connections with length-prefixed JSON framing
- NATTraversal: Hole-punching coordinator — STUN discovery, relay candidate exchange, direct QUIC with relay fallback
- WANDiscovery: Peer discovery via relay server, capability filtering, periodic re-registration
- WANManager: Central orchestrator with heartbeat/health loops, incoming offer handling, QUIC listener
- WANProvider: InferenceProvider conformance routing to WAN peers with 120s timeout

### Phase 4: Credit Economy
- Created CreditKit module (5 files) for token-based credit tracking
- CreditTypes: CreditAmount (Double wrapper), CreditTransaction, TransactionType, CreditPricing
- Pricing: (tokens/1K) * model_complexity * quant_multiplier; earners get 80% (20% network fee)
- CreditLedger: Actor with JSON persistence to ~/Library/Application Support/InferencePool/credits.json
- CreditWallet: @Observable wrapper for SwiftUI binding
- CreditMiddleware: CreditAwareProvider wrapping InferenceProvider for auto-tracking, balance checks
- CreditAnalytics: Daily/weekly/monthly summaries, top models/peers
- 100 credit welcome bonus for new users

### Phase 5: iOS Companion App
- Created SolairCompanion target (8 files) — thin client iOS app
- RemoteInferenceClient: URLSession HTTP client for Mac node API (streaming SSE + non-streaming)
- CompanionAppState: Bonjour discovery via NWBrowser, model list, conversation store
- TabView UI: Chat, Network, Wallet, Settings
- Only depends on SharedTypes — no MLX, no macOS-only APIs

### Integration
- Updated AppState with WANManager, CreditWallet, WAN toggle, async init
- Added WANView and WalletView to macOS app
- Updated sidebar with WAN + Wallet navigation items
- Updated DashboardView with WAN summary and credit balance sections
- Updated SettingsView with WAN config and credit summary
- Updated Package.swift: 11 targets, iOS 17 platform, 2 executables

## Key decisions
- Pure-Swift WAN networking (no rust-libp2p) — uses Network.framework's native QUIC
- CryptoKit Ed25519 for node identity (no external crypto deps)
- Relay server is signaling-only; actual inference data flows P2P via QUIC
- iOS app is thin client only — HTTP to Mac nodes, not full cluster protocol
- Credit pricing scales linearly with model size (0.1 credits per billion params per 1K tokens)
